    How Hush works

    Hush applies up to three independent layers of defense per site. Any single layer would help; combined they're airtight for the rules you've defined. Here's what each layer does and when to use it.

    1Block (network) — request is killed before it leaves the browser

    Patterns in the Block list register a rule in Chrome's declarativeNetRequest engine. When any request matches, the browser's network stack refuses it before DNS resolution, TCP connection, or TLS handshake. No bytes are sent to any server. The originating fetch() or iframe load fails locally with net::ERR_BLOCKED_BY_CLIENT.

    Rules are declared under a site in your config but applied as global URL-pattern matches. A rule under reddit.com blocks its target URL anywhere it's requested — including from third-party iframes embedded on Reddit (redgifs, YouTube, ad iframes, etc.). Chrome's initiatorDomains condition matches the initiating frame's origin, not the tab you're looking at, so using it would miss cross-origin iframe traffic. Per-site rule-authoring is preserved as your mental model, but execution is global by URL. If you need a URL blocked only on one site, make the pattern itself more specific.

    Blocking is most thorough at this layer — no script runs, no response is parsed, no iframe loads.

    Pattern syntax (uBlock-style):

    ||domain.com match requests to domain.com and any subdomain, across any path/port ||ads.example.com match requests to exactly ads.example.com or its subdomains ||tracker.net/pixel match requests whose path starts with /pixel *.cdn.example.com wildcard on subdomain (* = any characters) ^ separator anchor. Rarely needed after ||domain (the domain anchor already handles it) and can be problematic with hyphenated subdomains.

    When to use: whenever the request itself is what you want gone — tracking pixels, ad scripts, telemetry beacons, widget iframes, API calls to unwanted services.

    2Remove (DOM) — element is physically deleted

    Selectors in the Remove list are physically excised from the DOM via element.remove(). A MutationObserver watches every DOM change, so elements inserted later (by the site's SPA router, infinite scroll, modals, etc.) get removed as soon as they appear.

    This layer frees event listeners, breaks any internal references the element held, and stops anything that depends on the element being in the tree. Some JavaScript libraries voluntarily pause work when their target element is removed.

    Pattern syntax: any valid CSS selector.

    .modal-overlay
[data-role="promo"]
iframe[src*="tracker.example.com"]
#footer .subscribe-banner

    When to use: when you want the element completely gone — freeing listeners/timers tied to it, stopping libraries that animate it, or when the site re-adds the element and display: none isn't enough.

    3Hide (CSS) — elements still exist but are invisible

    Selectors in the Hide list are applied as a user stylesheet with display: none !important. The browser skips layout, paint, compositing, and GPU work for hidden elements. CSS animations and transitions on those elements also stop running (no rendered box = nothing to animate).

    This is the mildest layer. The element still exists in the DOM and its JavaScript can still execute. If the element's scripts poll a timer or make network requests, those keep running. Use Hide when the page will break if the element is physically missing (some sites crash when their own DOM is mutated) but you don't want it taking up screen space or GPU compositor cycles.

    Pattern syntax: any valid CSS selector (same as Remove).

    When to use: when something visual is annoying but removing the element might break the page (rare), or as a safety net for elements that might exist even after you've blocked their network resources.

    Behavioral suggestions (opt-in)

    Hush can watch a tab and suggest rules based on what the page does, surfacing the results in the popup for one-click acceptance. Turned off by default because it costs a small amount of per-scan CPU.

    What signals it uses (all observed locally in the current tab, no lists fetched):

    • sendBeacon targets - the navigator.sendBeacon API is purpose-built for telemetry. Any third-party host receiving beacons gets suggested as a block.
    • Tracking pixels - third-party <img> responses smaller than 200 bytes are the classic 1x1 tracking-pixel pattern.
    • First-party telemetry subdomains - subdomains of the current site whose responses are all tiny (< 1 KB) are almost always internal tracking/widget endpoints that curated lists can't catch (they're company-specific).
    • Polling endpoints - the same canonical URL fetched repeatedly in a short window with tiny responses is a heartbeat or telemetry loop.
    • Hidden iframes - iframes with display:none, visibility:hidden, 1x1 size, or positioned off-screen have no legitimate user-facing reason to render.
    • Sticky overlays - fixed-position elements with high z-index covering a significant portion of the viewport are usually nag popups or cookie banners.

    Why this catches what uBlock Origin Lite / AdGuard / Privacy Badger can't: curated blocklists only know about publicly-documented trackers. Hush looks at what this page is actually doing and flags per-site behaviors the lists have no way to know about (first-party telemetry subdomains, widget iframes, polling endpoints).

    Data egress: none. Hush does not fetch filter lists, does not send observations anywhere, and all analysis runs in the current tab's process. Suggestions are held in memory per-tab and cleared on navigation.

    Manual override: even with the feature off, the popup's Scan this tab now button runs a single scan on demand without enabling continuous scanning.

    Layer interaction and overlap
    • Hide and Remove the same selector: Remove runs first (on every mutation), so matching elements never stay long enough for Hide to count them. The popup will show such selectors as - (removed) in the Hidden section — that's not a bug, it's the honest story.
    • Block plus Hide: the request is killed at the network layer; if the site's JS still builds a DOM element (an iframe with a now-failed src, for example), Hide suppresses its rendering.
    • Domain matching for Hide/Remove selectors: a site entry for example.com also applies to www.example.com, m.example.com, and any other subdomain. Selectors execute only in frames whose hostname matches the site key (or one of its subdomains).
    • Block rules are global URL patterns: a pattern like ||ads.example.com under any site entry blocks that host wherever it's requested — in the top frame, in embedded iframes, from service workers, anywhere. Site-keyed authoring is a convention for organizing your rules, not a runtime scope.
    • Iframe-originated traffic is attributed to the tab: when a cross-origin iframe (e.g., an embedded redgifs video on Reddit) initiates a request, Hush reports it as part of the Reddit tab's activity. Your Reddit block rules cover iframe traffic automatically; you don't need per-iframe-domain rules.
    Allowlists (things the behavioral detector ignores)

    When the behavioral-suggestions detector would otherwise flag these as suspicious, it skips them instead. Default entries cover captcha, OAuth, payment iframes, and framework portal containers. Add your own site-specific entries as you encounter false positives.
